# SmetaOffice MVP Stabilization Checklist

Дата проверки: 2026-05-15  
Цель: стабилизировать MVP без загрузки приватных документов SmetaGroup в GitHub.

| Блок | Сценарий | Статус PASS/FAIL | Что проверено | Что сломано | Что исправить |
|---|---|---|---|---|---|
| Сборка | `npm ci` | PARTIAL | Первичная установка через стандартный `npm ci` упиралась в локальный `EPERM`/lock npm cache и занятые node-процессы. После остановки процессов зависимости восстановлены через локальный cache install. | Стандартный `npm ci` может падать на Windows, если сервер или Vite уже держит файлы Rollup/esbuild. | Перед чистым CI останавливать node-процессы; на сервере использовать чистую среду без lock-файлов. |
| Сборка | `npm run build` | PASS | Vite production build проходит, белого экрана на уровне сборки нет. | Не выявлено. | Повторять после каждой правки. |
| Запуск | `npm start` | PASS | Сервер стартует на `127.0.0.1:8787` и отдаёт приложение из `dist`. | Не выявлено. | Для онлайн-теста держать один активный серверный процесс. |
| API | `/api/health` | PASS | `/api/health` вернул `200` и JSON `{"ok":true,"service":"SmetaOffice API"}`. | Не выявлено. | Добавить версию приложения и дату последней записи БД в следующей итерации. |
| Авторизация | Все тестовые роли | PASS | Проверены входы: `owner`, `admin`, `deputy`, `director`, `regional`, `pm`, `project`, `sales`, `headsales`, `accountant`, `finance`, `executor`, `partner`. | Не выявлено. | В README держать только тестовые логины и пароли. |
| Права доступа | Владелец | PASS | `owner` видит весь холдинг, финансы, админку, исполнителей, партнёров и все разделы. | Не выявлено. | Дальше уточнить дашборд владельца по реальным KPI. |
| Права доступа | Администратор | PASS | `admin` видит админку, пользователей, исполнителей и проекты; финансовое меню скрыто. | Не выявлено. | Разделить системного администратора и CRM-администратора в следующей итерации. |
| Права доступа | Заместитель | PASS | `deputy` видит операционный контур и финансы как управленческая роль. | Не выявлено. | Уточнить, какие заместители имеют финансы, а какие только производство. |
| Права доступа | Руководитель направления | PASS | `director` фильтруется по своему направлению/регионам, не получает админку. | Требуется ручная проверка на большем наборе проектов. | Добавить реальные назначения `directorUserId` при создании проектов. |
| Права доступа | Региональный менеджер | PASS | `regional_manager` фильтруется по своему региону, не получает финансы и админку. | Требуется ручная проверка с несколькими регионами. | Добавить обязательный регион у каждого проекта и пользователя. |
| Права доступа | РП / менеджер проекта | PASS | `pm` и `project_manager` видят назначенные проекты; логика допуска учитывает `pmUserId`, `projectManagerId`, `managerId`. | В форме создания проекта пока нет выбора конкретного РП/менеджера. | Добавить назначение РП/менеджера при создании/редактировании проекта. |
| Права доступа | Продажи | PASS | `sales_manager` и `head_of_sales` не видят финансы/исполнителей; доступ ограничен клиентским и продажным контуром. | Hunter/Farmer пока не выделены как отдельные сущности сделки. | Добавить модель лида: Hunter, Farmer, РОП, источник, SLA ответа. |
| Права доступа | Финансы и бухгалтерия | PASS | `finance` и `accountant` видят финансовый раздел и денежные таблицы, но не видят админку. | Нет отдельной выгрузки под бухгалтерию/акты. | Добавить экспорт/отчёт по оплатам, актам, долгам и выплатам. |
| Права доступа | Исполнитель | PASS | `executor` видит только задачи/этапы, где назначен его `executorId`; меню финансы/админка/исполнители скрыто. | Если в проекте нет назначенных задач, экран может выглядеть пустым. | Добавить понятный empty-state "нет назначенных работ". |
| Права доступа | Партнёр | PASS | `partner` видит только назначенные работы/проекты и не видит чужую клиентскую базу, финансы и исполнителей. | Партнёрская модель пока неполная. | Добавить карточку партнёра: тип, регион, направление, заявки, выплаты, просрочки, история. |
| Проекты | Список и открытие проекта | PASS | Проект можно открыть из списка; карточка проекта показывает этапы, задачи, документы, чат и финансы по правам. | Не все кнопки являются боевыми действиями; часть показывает уведомление MVP. | Разделить реальные действия и демонстрационные подсказки. |
| Проекты | Создание проекта | PARTIAL | В коде есть форма создания проекта с регионом, направлением, типом, клиентом и папкой Яндекс.Диска; создаются шаблонные этапы. | Нужна дополнительная ручная UI-проверка полного сценария после перезапуска сервера. | Проверить руками: создать проект, обновить страницу, открыть его, перезапустить сервер. |
| Проекты | Сохранение JSON | PASS | `server.mjs` сохраняет `/api/projects`, `/api/executors`, `/api/users` в `data/database.json`; локально файл исключён из Git. | JSON-БД подходит только для MVP и одного сервера. | Для боевого теста перейти на Supabase/PostgreSQL или аналог. |
| Этапы | Автошаблоны по типу проекта | PASS | Есть шаблоны: дизайн-проект, архитектурный проект, проектная документация по 87 постановлению, обследование/ТЗК, ремонт, недвижимость, комплектация. | Нет UI редактирования всех финансовых полей этапа в одном месте. | Сделать редактор этапа: исполнитель, срок, процент, суммы, документы, комментарии. |
| Этапы | Поля этапа | PASS | У этапа есть исполнитель, `executorId`, срок, статус, процент, ссылка Яндекс.Диска, документы, комментарии, цена клиента, стоимость исполнителя, выплачено, остаток, финстатус. | Часть полей создаётся с нулями и не всегда редактируется через UI. | Добавить полноценное редактирование этапа и историю изменений. |
| Исполнители | Добавление исполнителя | PASS | Модуль исполнителей позволяет добавить исполнителя, заполнить тип, город, специализацию, ранг, статус, контакты и заметку. | Не проверена массовая загрузка из Excel. | Добавить импорт из Excel через backend и маппинг колонок. |
| Исполнители | Редактирование и загрузка | PASS | В карточке исполнителя есть рейтинг, ранг, загрузка, начислено/выплачено/к выплате, активные задачи. | Финансы исполнителя пока тестовые и считаются не полностью от этапов. | Связать начисления исполнителя с фактическими этапами и выплатами. |
| Исполнители | Контакты | PASS | Контакты скрываются для ролей без `viewExecutorContacts`; партнёр не видит чужую базу. | Текст карточки ещё говорит "владелец", хотя право есть у нескольких управленческих ролей. | Уточнить текст: "уполномоченные роли". |
| Финансы | Поля проекта | PARTIAL | Есть договор, факт оплат, дебиторка, бюджет реализации, расходы, выплаты партнёрам, операционные/зарплатные расходы, прибыль, маржинальность, 67/33. | Нет отдельного поля "статус оплаты" в финансовой таблице; не все поля редактируются из UI. | Сделать финансовую карточку проекта с редактированием и журналом. |
| Финансы | Поля этапа | PARTIAL | В модели этапа есть клиентская цена, стоимость исполнителя, оплачено, остаток и финстатус. | Нет полноценного сценария оплаты этапа. | Добавить статусы оплаты этапа: план, счёт, оплачено, к выплате, выплачено. |

| Продажи | Модель Lead / SalesLead | PASS | Модель лида сохранена: источник, регион, направление, клиент, бюджет, стадия, Hunter, Farmer, РОП, партнёр, проект, SLA, история. Автоподмешивание демо-лидов отключено. | Реальный Bitrix24 API не подключён намеренно. | Подключать API только отдельным согласованным шагом. |

| Продажи | Связь Lead → Project | PASS | Лид можно связать с существующим проектом SmetaOffice, в карточке отображается `projectId`. | Автоматическое создание проекта из сделки пока не включено. | Добавить создание проекта только после договора/аванса. |
